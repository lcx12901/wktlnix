# 子代理策略

核心指令：任何非简单对话消息都应生成子代理。

## 什么时候使用子代理

使用子代理用于：

- 搜索（网络、社交、电子邮件）
- API调用
- 多步骤任务
- 数据处理
- 超出简单读取的文件操作
- 日历/电子邮件操作
- 任何预期需要超过几秒钟的任务
- 任何可能导致失败或阻塞主会话的内容

## 何时直接处理

在不使用子代理的情况下处理以下内容：

- 简单的对话回复
- 快速澄清问题
- 确认
- 快速读取文件以获取上下文
- 单步查找，其中创建子代理的时间将比直接完成更长

目标是保持主会话的响应性，而不是为了创建子代理而创建。如果直接方法更快、更简单，就使用它。

## 编程、调试和调查委托

所有编程、调试和调查任务都通过子代理进行。主会话永远不应该被这项工作阻塞。

子代理评估复杂性：

- **简单**：直接处理。配置更改、小的单文件修复、追加到现有模式、检查一个日志或配置值。
- **中等 /主要**：
  委托给你的编程代理CLI。这包括多文件功能、复杂逻辑、大量添加、以及需要跨文件
  或系统进行跟踪的多步骤调查。

## 为什么

主会话的稳定性至关重要。子代理：

- 保持主会话的响应性，以便用户可以继续交谈
- 将故障与主对话隔离
- 允许多任务并行处理
- 完成后报告

## 授权公告

当授权给子代理时，告诉用户你正在使用哪个模型和提供者。这使路由变得可见。
格式：[模型] 通过 [提供者/工具] 示例：

- 使用 <模型> 创建子代理来搜索推特。
- 授权给 <编码模型> 通过编码代理CLI。

## 错误处理

当子代理失败时：

1. 通过消息平台向用户报告错误详情
2. 如果失败似乎是暂时的（网络超时、速率限制），重试一次
3. 如果重试仍然失败，报告两次尝试并停止

## 实施

使用您的框架的子代理生成机制，包括：

- 清晰的任务描述
- 对于非编码子代理任务，默认使用主要模型
- 只有在主要模型不可用或任务需要特殊能力（例如，特定的 API
  访问）时，才使用不同的模型
- 如有帮助，请提供估计时间
